# 题目
基于智能体的自然语言智能家庭协同控制系统的设计与实现

# 摘要

硬件部分有以运行 Arduino 的 ESP32 家族为基础的传感器（如温湿度、霍尔、空气质量、光线、人体移动、人在等）和触发器（如机械臂、指示灯、显示屏、音响等），他们与服务端进行 MQTT 通信，在设计中触发器可以采用仅设计到 IO 输出。

软件部分为后端 Bun + AI-SDK + MQTT + Redis + TypeORM(PgSQL)，前端 React + AI-SDK + Tailwind + Vite + swr + TanStack Router + Zustand + shadcn。

协同工作主要是有如下几点：
- 通信系统中，基于 MQTT 对通信进行规范
- 软件系统中，基于面向对象的设计思想，在传感器和触发器基类之上，封装每个设备类型的类，其中包括任何可读取的属性、可调用的方法、可被调用的回调函数（触发器）
- 硬件系统中，根据软件系统的设计，完善相关硬件设计
- 全局中，每个设备基类应该有如下层级标签：家庭 -> 房间，用于标识每个设备的所在位置，且应该有设备类型、设备 ID、设备名称等标签，方便用户进行语音操作

需求：
- 用户旅程上有家庭、房间、设备的层级概念，房间中可以手动添加/绑定设备
- 语言模型 + Function Call 在家庭层级，每个家庭的上下文对于语言模型应当是独立的
- 语言模型应当分为后台服务（用于处理传感器的触发事件）和前台服务（用户的文字/语言需求），且互不干扰
- 注册了的触发器会携带参数调用语言模型的后台服务，同时因为设备属性也是定期上报，在节约 token 的前提下，定期处理（如温湿度上报联动是否要开启空调、加湿器、除湿器）
- 同时还有用户自行配置的自动化，这些自动化应独立于语言服务，因为其是确定性的，且应该方便使用 JSON 生成
- 用户通过前台语言模型交互应该可以使用自然语言创建自动化
- 前台语言模型也应该响应用户的操作或读取设备状态的需求
- 可预设典型的智能体驱动全屋自动化案例（如房间温湿度+季节+当天天气联动空调/加湿器/除湿器），并可通过配置开关控制是否启用
