# 架构设计

## 目标与范围
- 覆盖软硬件协同的系统形态：设备层（ESP32/传感器/触发器）、通信层（MQTT Broker）、后端（Node.js + Express + Redis + PostgreSQL + AI-SDK）、前端（React）。
- 明确 MQTT 协议/主题规范，设备注册、命令下发、数据上报、事件回调的流程。
- 设计数据库实体，支撑家庭/房间/设备/自动化/日志等核心场景。
- 预留可扩展性（新增设备类型、自动化模板、外部数据源）与可观测性。

## 系统分层
- 设备侧：ESP32 运行 Arduino，封装传感器/触发器；本地维持心跳/断线重连/简易队列。
- 通信层：MQTT Broker（QoS1/遗嘱/ACL），区分注册、控制、上报、事件。
- 后端：Node.js API（鉴权、路由、设备管理、自动化调度）、后台模型（处理设备事件）、自动化引擎（确定性 JSON 规则）、Redis（缓存/队列/速率）、PostgreSQL（持久化）。
- 前端：React + AI-SDK，提供家庭/房间/设备管理、自动化配置、语音/文本控制；前台模型按家庭隔离上下文。

## MQTT 协议与主题设计
- 客户端标识：`clientId = home/{homeId}/room/{roomId}/device/{deviceId}`；用户名为 `deviceId`，密码使用一次性注册 token 或设备密钥。
- 主题前缀：`home/{homeId}/room/{roomId}/device/{deviceId}`；事件型主题附加语义后缀。
- 主要主题
  - 注册：设备主动发布 `.../register`（QoS1），负载含设备指纹；后端在 `.../register/ack` 回复凭据/配置。
  - 心跳：设备定期发布 `.../lwt`（或保活 ping），Broker 配置遗嘱消息在断线时发布到 `.../status`，状态 `offline`。
  - 状态/属性上报：周期或事件触发发布 `.../telemetry`；示例负载：
    ```json
    {
      "ts": 1700000000,
      "attrs": {"temperature": 23.5, "humidity": 58},
      "meta": {"fw": "1.0.3", "battery": 92}
    }
    ```
  - 事件上报：`.../event/{eventType}`（如 `motion`, `door`），负载携带事件参数与时间戳。
  - 命令下发：后端向 `.../command` 发布（QoS1）；负载示例：
    ```json
    {
      "cmdId": "uuid",
      "method": "set_ac",
      "params": {"mode": "cool", "temp": 25},
      "timeout": 5000
    }
    ```
  - 命令回执：设备向 `.../command/ack` 回复结果：
    ```json
    {"cmdId": "uuid", "status": "ok", "result": {"temp": 25}}
    ```
- 消息格式：统一 JSON，必须包含 `ts`（服务器或设备时间）。使用 QoS1 对关键链路（注册、命令、事件），QoS0 对高频遥测可选。编码 UTF-8。
- 安全：Broker 侧 ACL 按 `home/+/room/+/device/{deviceId}/#` 限定；注册后下发持久凭据；TLS 可选。

## 设备生命周期流程
1) 预注册：后端生成设备密钥/一次性 token，与设备 ID/类型/所属房间绑定。
2) 连接：设备携带 clientId/用户名/密码接入 MQTT，Broker 校验。
3) 注册上报：设备发 `.../register`，包含硬件信息、固件版本、能力列表（属性/方法/事件）。
4) 后端校验与应答：在 `.../register/ack` 返回：
   - 成功：返回 `status=ok`，分配心跳周期、期望上报字段、命令能力、时间同步信息。
   - 失败：`status=error` 与原因。
5) 心跳与 LWT：设备按配置周期上报；断线由遗嘱标记 `offline`。
6) 遥测/事件上报：按策略（周期/变化/阈值）推送。
7) 命令下发与回执：后端投递命令，设备执行并回执；超时由后端标记失败。
8) OTA/配置更新（预留）：通过 `.../config` 或 `.../ota` 下发。

## 数据库实体设计（PostgreSQL）
- `users`：用户账号，含 hashed 密码/第三方登录、角色。
- `homes`：家庭，归属用户，包含名称/地址/时区。
- `rooms`：房间，归属家庭，名称/楼层/类型。
- `devices`：设备主表，归属房间；字段含 `device_id`（物理 ID）、`name`、`type`、`category`（sensor/actuator/both）、`status`（online/offline）、`fw_version`、`secret`、`last_seen`.
- `device_capabilities`：能力声明，`device_id` 外键，字段 `kind`（attr/method/event）、`name`、`schema`（JSON）用于描述参数/单位/范围。
- `device_attrs_snapshot`：设备最新属性快照，JSONB 存储。
- `telemetry_logs`：属性/遥测历史，含 `home_id/room_id/device_id/ts/payload`（JSONB）。
- `device_events`：事件日志，含事件类型、参数、触发来源。
- `commands`：命令下发记录，含 `cmd_id`、`device_id`、`method`、`params`、`status`、`sent_at`、`ack_at`、`error`.
- `automations`：自动化规则，归属家庭；字段 `name`、`enabled`、`source`（preset/nl/json）、`definition`（JSON，DSL 或条件/动作结构）、`scope`（room/device/all）。
- `automation_runs`：自动化执行记录，含输入、决策、输出指令、结果。
- `llm_sessions`：前台/后台模型会话元信息（按家庭隔离），含 `role`（front/back）、`context_ref`。
- `llm_invocations`：模型调用日志，含 token 成本、输入摘要、输出摘要、关联的事件/用户请求。
- `external_data_cache`：外部数据（天气/季节）缓存，含数据源、有效期。
- `audit_logs`：鉴权/管理操作审计。

## 自动化与模型分工
- 后台模型：订阅设备事件/遥测，参与判定是否需要触发动作；以节流策略减少调用频率。
- 自动化引擎：优先使用确定性 JSON 规则执行（条件-动作），独立于 LLM；LLM 可作为规则生成/建议。
- 前台模型：服务用户查询/控制/创建自动化，按家庭隔离上下文。
- 触发器联动：注册触发器调用后台模型或直接触发自动化；动作经命令链路下发。

## Redis 使用
- 会话与上下文缓存（前后台模型）。
- 速率限制与节流（设备上报、外部数据获取）。
- 短期队列（待下发命令/事件桥接）。
- 分布式锁（避免重复执行自动化）。

## 安全与可观测性
- 鉴权：API 使用 JWT/OAuth；MQTT 设备使用设备密钥/一次性 token 注册后换长效密钥。
- 隔离：家庭级 ACL；模型上下文按家庭分区。
- 日志/指标：记录设备上下行、命令成功率、LLM 成本、自动化命中率；暴露 Prometheus 指标。
- 审计：关键操作写 `audit_logs`，模型调用写 `llm_invocations`。

## 开发优先级（落地顺序）
1) 定义 MQTT 主题/消息格式 + Broker ACL/LWT 配置；完成设备注册/心跳/基础上报/命令闭环。
2) 抽象设备模型（能力声明）与数据库基础表（homes/rooms/devices/device_capabilities/telemetry_logs/commands）。
3) 后端 API：家庭/房间/设备管理，命令下发与回执持久化。
4) 自动化引擎最小可用（JSON 规则，条件-动作），后台模型订阅事件接入。
5) 前端：家庭/房间/设备管理界面 + 文本/语音控制 + 自动化配置。
6) 可观测性与外部数据接入（天气/季节），优化模型成本与节流策略。
