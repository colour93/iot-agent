# 项目梳理（2026-03-12）

## 1. 仓库结构

- `device-firmware/esp32-family`：ESP32 固件（PlatformIO + Arduino）。
- `server/backend`：Node.js + Express + TypeORM + MQTT + Redis + PostgreSQL 的后端服务。
- `server/frontend`：React + Vite + TanStack Router + SWR + Zustand 前端控制台。
- `docs`：需求、架构、MQTT 协议等设计文档。

## 2. 端到端链路

1. 设备侧（ESP32）连接 Wi-Fi 和 MQTT，按 `device/{deviceId}/...` 主题发布：
   - `register`（能力上报）
   - `telemetry`（属性上报）
   - `lwt/status`（心跳与在线状态）
   - `command/ack`（命令回执）
2. 后端 MQTT 服务订阅上述主题，写入 PostgreSQL，更新设备状态与快照。
3. 前端通过 `/api/*` 调用后端：
   - 家庭/房间/设备管理
   - 命令下发
   - 自动化管理
   - LLM 对话
4. 自动化引擎根据 JSON 规则判定触发动作，动作可下发设备命令。

## 3. 后端模块职责（`server/backend/src`）

- `app.ts`：启动入口，初始化 Postgres、Redis、MQTT、路由与错误处理中间件。
- `routes/`：
  - `authRoutes.ts`：登录/注册与 JWT 签发。
  - `homeRoutes.ts`：家庭与房间管理。
  - `deviceRoutes.ts`：设备预注册、查询、命令下发、属性快照查询。
  - `automationRoutes.ts`：自动化 CRUD 与手动触发。
  - `llmRoutes.ts`：前台/后台 LLM 接口与 chat 接口。
  - `metricsRoutes.ts`、`pingRoutes.ts`：健康与基础指标。
- `services/`：
  - `mqttService.ts`：MQTT 收发与消息处理。
  - `automationEngine.ts`：确定性规则执行。
  - `commandService.ts`：命令超时扫描与重试。
  - `llmService.ts`：LLM 会话与工具调用。
  - `redisClient.ts`：Redis 客户端初始化。
- `entities/`：家庭、房间、设备、命令、自动化、LLM 调用记录等数据模型。

## 4. 前端模块职责（`server/frontend/src`）

- 路由页面：
  - `/`：家庭、房间、设备总览与设备预注册/命令下发。
  - `/automations`：自动化规则展示与 JSON 编辑保存。
  - `/chat`：前台模型对话。
  - `/observability`：观测页面（当前以静态演示数据为主）。
  - `/login`：登录/注册。
- 数据层：
  - `lib/api.ts`：API 请求封装。
  - `lib/swr-hooks.ts`：SWR 拉取 homes/rooms/devices/automations。
  - `lib/store.ts`：全局状态（token、选中 home/room、聊天记录）。

## 5. 固件侧实现（`device-firmware/esp32-family`）

- `src/main.cpp`：
  - 连接 Wi-Fi 和 MQTT。
  - 启动后发布注册消息。
  - 周期发布心跳与遥测。
  - 订阅 `command` 执行并回复 `command/ack`。
- `include/consts.h`：网络、MQTT、设备标识与周期配置。

## 6. 当前实现与设计文档差异

- 文档主张主题层级为 `home/{homeId}/room/{roomId}/device/{deviceId}`，当前代码实际使用 `device/{deviceId}/...`。
- 后端已做 JWT 鉴权与基础路由保护，但家庭级授权边界仍不完整（`homeIds` 未形成闭环）。
- 自动化与 LLM 功能链路存在，但仍有若干类型与签名问题导致构建失败（见第 7 节）。

## 8. 编码与文件规范（UTF-8）

- 已对源码/配置/文档（排除 `.git`、`.pio`、`node_modules`）做严格 UTF-8 解码检查。
- 检查结果：`checked=128`，全部可按 UTF-8 严格解码。
- 后续读写建议统一使用 UTF-8（无 BOM）。
