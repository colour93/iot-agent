# 项目梳理（2026-03-12）

## 1. 仓库结构

- `device-firmware/esp32-family`：ESP32 固件（PlatformIO + Arduino）。
- `server/backend`：Bun + Express + TypeORM + MQTT + Redis + PostgreSQL 后端服务。
- `server/frontend`：React + Vite + TanStack Router + SWR + Zustand 前端控制台。
- `docs`：需求、架构、MQTT 协议等文档。

## 2. 端到端链路

1. 设备连接 Wi-Fi 与 MQTT，按 `device/{deviceId}/...` 主题上报：
   - `register`：能力注册
   - `telemetry`：属性遥测
   - `event/{eventType}`：设备事件
   - `lwt/status`：在线状态
   - `command/ack`：命令回执
2. 后端 MQTT 消费消息并写入 PostgreSQL，更新设备状态/快照/日志。
3. 前端通过 `/api/*` 调用后端，完成家庭、房间、设备、自动化、LLM 对话等操作。
4. 后端自动化引擎基于规则触发动作，可下发设备命令或执行通知类动作。

## 3. 后端模块职责（`server/backend/src`）

- `app.ts`：服务启动入口，初始化 Postgres、Redis、MQTT、路由、错误处理中间件与命令超时任务。
- `config/env.ts`：读取 `config.yaml`，合并默认值并做严格 schema 校验。
- `db/data-source.ts`：TypeORM 数据源初始化。
- `routes/`
  - `authRoutes.ts`：登录/注册、JWT 签发。
  - `homeRoutes.ts`：家庭与房间管理。
  - `deviceRoutes.ts`：设备列表、预注册、命令下发、属性快照查询。
  - `automationRoutes.ts`：自动化 CRUD 与手动触发。
  - `llmRoutes.ts`：LLM front/back/chat 接口。
  - `metricsRoutes.ts`、`pingRoutes.ts`：健康检查与基础状态。
- `services/`
  - `mqttService.ts`：MQTT 连接、主题订阅、消息处理。
  - `commandService.ts`：命令超时扫描与重试。
  - `automationEngine.ts`：自动化规则执行。
  - `llmService.ts`：LLM 调用与工具调用。
  - `redisClient.ts`：Redis 连接初始化。
- `entities/`：家庭、房间、设备、命令、自动化、遥测、LLM 会话/调用等模型。

## 4. 后端流程（优化梳理版）

1. 启动阶段（`app.ts`）
   - 加载并校验配置（`config/env.ts`）。
   - 初始化 Postgres（`initDataSource`）。
   - 初始化 Redis（`initRedis`）。
   - 启动 MQTT 客户端并订阅设备主题（`startMqttService`）。
   - 注册路由与统一错误处理。
   - 启动命令状态维护任务：
     - 每 5 秒扫描 `sent -> timeout`
     - 每 10 秒对 `timeout` 进行有限重试

2. MQTT 入站链路（`mqttService.ts`）
   - `register`：校验设备是否预注册，更新设备状态与能力，回发 `register/ack`。
   - `telemetry`：更新 `Device.lastSeen/status`，更新或创建 `DeviceAttrsSnapshot`，写入 `TelemetryLog`。
   - `event/{eventType}`：写入 `DeviceEvent`。
   - `command/ack`：更新 `Command` 状态为 `acked/failed`。
   - `lwt/status`：更新设备在线状态与最后活跃时间。

3. API 命令下发链路（`deviceRoutes.ts` + `mqttService.ts`）
   - `POST /api/devices/:deviceId/command`
   - 路由层做设备存在性与基础权限校验。
   - 组装命令并发布到 `device/{deviceId}/command`（QoS 1）。
   - 后续由设备 `command/ack` 回执闭环状态。

4. 自动化执行链路（`automationRoutes.ts` + `automationEngine.ts`）
   - 支持手动触发与规则触发。
   - `evaluateAutomation` 按 `conditions -> actions` 执行。
   - `action.kind === command` 时落库 `Command` 并发布 MQTT 命令。
   - 运行状态写入 `AutomationRun`（`running/succeeded/failed`）。

5. LLM 链路（`llmRoutes.ts` + `llmService.ts`）
   - `front/back`：记录会话与调用日志（当前是轻量实现）。
   - `chat`：先做 Redis 限流（每家庭每分钟最多 20 次），再调用 AI SDK `generateText` 与工具。
   - 工具能力包括：设备列表、房间列表、设备命令下发。

6. 可观测性链路
   - `GET /healthz`：进程健康。
   - `GET /api/ping/redis`：Redis 连通性。
   - `GET /api/metrics/mqtt`：MQTT 连接状态和最近错误。

## 6. 当前实现与设计文档差异

- 主题层级当前使用 `device/{deviceId}/...`，并未采用 `home/{homeId}/room/{roomId}/device/{deviceId}`。
- JWT 鉴权链路已接入，但家庭级授权边界仍需进一步闭环（`homeIds` 目前未完整贯通）。
- LLM 侧已具备工具调用骨架，但还需继续做模型配置标准化与生产级异常治理。

## 7. 后续优化优先级（建议）

1. 完成 `homeIds` 的登录态注入与全链路 home 级权限校验。
2. 在命令下发接口补齐命令落库，统一“下发-回执-超时-重试”追踪闭环。
3. 收敛生产环境配置：`jwt.secret` 强化、`cors` 白名单化、LLM 模型 ID 规范化。

## 8. 编码与文件规范（UTF-8）

- 源码、配置、文档建议统一 UTF-8（无 BOM）。
- 中文内容编辑时固定 UTF-8，避免终端查看出现乱码但文件本体可编译/可解析的情况。
