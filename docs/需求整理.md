# 需求整理

## 系统定位
- 面向家庭场景的智能体驱动自然语言协同控制系统，连接传感器、触发器与前后端应用，实现语音/文本与自动化联动。

## 硬件需求
- 设备：基于运行 Arduino 的 ESP32 系列，包含传感器（温湿度、霍尔、空气质量、光照、人体移动/存在等）与触发器（机械臂、指示灯、显示屏、音响等，至少需 IO 输出能力）。
- 通信：设备通过 MQTT 与服务端交互，需支持设备状态上报、指令下发与事件回调。
- 标识：设备具备层级标签（家庭 -> 房间）、设备类型、设备 ID、设备名称，便于语音/自然语言定位控制。

## 软件需求
- 设备：ESP32 + Arudino + PsychicMqttClient + PlatformIO
- 后端：Node.js + AI-SDK + MQTT + Redis + PostgreSQL；负责设备管理、消息路由、自动化执行与模型调用。
- 前端：React + AI-SDK + Tailwind + Vite + SWR + TanStack Router + Zustand + shadcn；提供家庭/房间/设备管理、自然语言交互与自动化配置界面。
- 通信规范：在软件层基于 MQTT 定义主题/消息格式，区分设备上报、指令、事件回调等类型。
- 设备抽象：以面向对象设计封装传感器/触发器基类，派生具体设备类型，包含可读属性、可调用方法、可注册的回调。

## 功能需求
- 家庭与房间管理：支持创建家庭、房间；在房间中手动添加/绑定设备。
- 自然语言控制：用户可通过前台语言模型读取设备状态、下发控制指令。
- 自动化创建：用户可用自然语言创建自动化规则；同时支持 JSON 方式配置确定性自动化。
- 上下文隔离：语言模型在家庭维度隔离上下文，每个家庭拥有独立对话/状态。
- 前后台模型分离：后台模型处理传感器触发事件；前台模型处理用户文本/语音需求，二者互不干扰。
- 触发器联动：注册的触发器携带参数调用后台模型；设备属性定期上报，后台在节约 token 前提下周期性判断联动（如温湿度联动空调/加湿器/除湿器）。
- 预设自动化案例：提供典型的智能体驱动全屋自动化模板（如房间温湿度 + 季节 + 当天天气联动空调/加湿器/除湿器），可配置开关启停。
- 自动化执行：用户自定义自动化独立于语言服务，确定性执行；需易于通过 JSON 描述和持久化。

## 数据与存储
- Redis：会话/上下文缓存、消息队列或速率控制。
- PostgreSQL：家庭/房间/设备、自动化规则、用户信息等持久化。
- MQTT：消息通道，需支持 QoS/遗嘱/保活等基本特性。

## 非功能需求
- 可扩展性：便于新增设备类型与方法；MQTT 主题/消息结构可演进。
- 隔离性：家庭级上下文与控制隔离，防止跨家庭误操作。
- 成本控制：后台模型处理需控制 token 消耗，策略化采样/节流。
- 可观测性：设备状态上报与模型调用需记录日志/指标，便于调试。

## 开发优先级建议
1) 定义 MQTT 主题与消息格式；实现设备注册/认证/心跳与基本上报/下发。
2) 抽象设备基类与传感器/触发器派生，实现最小闭环（示例：温湿度传感器 + 指示灯/音响触发器）。
3) 后台模型工作流：订阅设备事件 -> 规则/模型判定 -> 下发控制。
4) 前台模型与前端：家庭/房间/设备管理界面 + 文本/语音控制 + 自动化自然语言创建。
5) 自动化引擎：JSON 规则存储、调度与执行，支持与模型分离的确定性链路。
